v3.0.0

Released on January 15, 2024

Major Release Breaking Changes Security Update

🎉 Highlights

Neo Service Layer v3.0.0 is our biggest release yet, introducing Intel SGX support for hardware-based security, multi-blockchain architecture improvements, and significant performance enhancements across all services.

✨ New Features

  • Intel SGX Integration: Hardware-based security enclaves for cryptographic operations and sensitive data protection

    #1234 • @security-team • Documentation

  • Multi-Region Failover: Automatic failover across AWS, Azure, and GCP with sub-15-minute RTO

    #1189 • @infrastructure-team • Documentation

  • GraphQL API: New GraphQL endpoint with subscription support for real-time blockchain data

    #1156 • @api-team • Documentation

  • Advanced Analytics Engine: Real-time analytics with ML-powered anomaly detection and predictive insights

    #1098 • @analytics-team

🚀 Improvements

  • Performance: 3x faster transaction processing with new parallel validation engine

    #1245 • Benchmark: 10,000 TPS achieved

  • Memory Usage: Reduced memory footprint by 40% through improved caching strategies

    #1223 • @performance-team

  • SDK Bundle Size: JavaScript SDK reduced by 55% with tree-shaking and code splitting

    #1201 • Now only 89KB gzipped

🐛 Bug Fixes

  • Fixed race condition in WebSocket connection pooling that could cause duplicate event emissions

    #1267 • Reported by @user123

  • Resolved memory leak in transaction validation service during high-load scenarios

    #1254 • Critical fix

  • Fixed incorrect gas estimation for complex smart contract interactions

    #1248

⚠️ Breaking Changes

  • API Authentication: JWT tokens now require RS256 algorithm. HS256 tokens are no longer accepted.

    Migration Guide • 6-month deprecation notice given

  • SDK Method Names: Renamed getBlockByHash() to block.getByHash() for consistency

    Automated migration tool available

  • Database Schema: Updated transaction table structure for improved indexing performance

    Migration script included in release

🔒 Security Updates

  • CVE-2024-0001: Fixed potential XSS vulnerability in transaction memo field rendering

    Severity: Medium • CVSS: 6.1 • Details

  • Updated all dependencies to latest secure versions, including critical OpenSSL update

    #1290

Migration Guide to v3.0.0

Step 1: Update Authentication 
// Before (v2.x)
const client = new NeoServiceLayer({
    auth: {
        algorithm: 'HS256',
        secret: process.env.JWT_SECRET
    }
});

// After (v3.0)
const client = new NeoServiceLayer({
    auth: {
        algorithm: 'RS256',
        publicKey: process.env.JWT_PUBLIC_KEY,
        privateKey: process.env.JWT_PRIVATE_KEY
    }
});
Step 2: Update SDK Method Calls 
// Before (v2.x)
const block = await client.getBlockByHash(blockHash);
const tx = await client.getTransactionByHash(txHash);

// After (v3.0)
const block = await client.block.getByHash(blockHash);
const tx = await client.transaction.getByHash(txHash);

// Automated migration available:
npx @neo-service-layer/migrate-v3
Step 3: Database Migration 
# Backup your database first!
pg_dump -h localhost -U neo -d neo_production > backup_v2.sql

# Run migration script
./migrations/v3.0.0/migrate-database.sh

# Verify migration
psql -U neo -d neo_production -c "SELECT version FROM schema_migrations;"

Estimated migration time: 2-4 hours for typical deployments

Support: Contact support@neoservicelayer.com for migration assistance

Versioning Policy

Semantic Versioning

Neo Service Layer follows Semantic Versioning 2.0.0 with the format MAJOR.MINOR.PATCH:

  • MAJOR version increments indicate incompatible API changes
  • MINOR version increments add functionality in a backwards compatible manner
  • PATCH version increments make backwards compatible bug fixes
Version Format Examples 
3.0.0     # Major release with breaking changes
2.5.0     # Minor release with new features
2.4.3     # Patch release with bug fixes
3.0.0-rc1 # Release candidate
3.1.0-beta.2 # Beta release

Deprecation Policy

Deprecation Process

Deprecation Announcement

Feature marked as deprecated in release notes and documentation. Console warnings added to SDK.

6-Month Grace Period

Feature remains functional with deprecation warnings. Migration guides and tools provided.

Final Warning (30 days)

Final reminder sent to all users. Enhanced warnings in logs and dashboards.

Feature Removal

Deprecated feature removed in next major version. Clear error messages guide to alternatives.

Currently Deprecated Features
  • client.getLegacyBlock() - Use client.block.get() instead
  • HS256 JWT algorithm - Migrate to RS256 by v4.0.0
  • REST API v1 endpoints - Use v2 endpoints with improved performance

Release Cycle

Release Schedule

Major Releases

Every 12-18 months

  • New features and improvements
  • Potential breaking changes
  • Performance optimizations
  • Architecture updates

Minor Releases

Every 2-3 months

  • Backwards-compatible features
  • Non-breaking improvements
  • New API endpoints
  • SDK enhancements

Patch Releases

As needed

  • Bug fixes
  • Security patches
  • Performance fixes
  • Documentation updates

Release Process

  1. Planning: Features planned and discussed in public roadmap
  2. Development: Implementation in feature branches
  3. Testing: Comprehensive testing including:
    • Unit tests (>95% coverage)
    • Integration tests
    • Performance benchmarks
    • Security audits
  4. Beta Release: Public beta for community testing
  5. Release Candidate: Final testing phase
  6. General Availability: Production release
  7. Post-Release: Monitor for issues and gather feedback
